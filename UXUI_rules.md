# FCM UX/UI Operating Rules

## 0) One-line Definition
FCM UX/UI의 목적은 보기 좋은 화면이 아니라, 복잡한 업무를 더 빠르고 정확하게 끝내게 만드는 것이다.

## 1) Core Axes (Always Validate)
모든 화면은 아래 5축으로 검토한다.

1. Context: 사용자가 현재 대상, 상태, 다음 행동을 즉시 파악할 수 있는가
2. State-Driven Action: 상태가 가능한 행동과 편집 범위를 결정하는가
3. Execution Surface: 목록과 상세가 판단/실행을 지원하는가
4. Permission Clarity: 권한 차이가 일관되고 예측 가능하게 보이는가
5. Operational Efficiency: 반복 업무에서 클릭/이동/입력이 줄어드는가

## 2) Must Rules (Non-negotiable)

### 2.1 Goal and Flow
1. 화면은 기능 나열이 아니라 사용자 작업 흐름 중심으로 설계한다.
2. 한 화면의 핵심 목표는 하나로 정의한다.
3. 화면 전환 후에도 작업 맥락이 유지되어야 한다.

### 2.2 Layout and Navigation
1. 데스크톱은 복합 업무 처리를 위한 기본 환경으로 설계한다.
2. 탐색 영역, 작업 영역, 보조 정보 영역의 역할을 분리한다.
3. 리스트, 상세, 협업 정보가 함께 필요한 화면은 맥락 유지형 구조를 우선한다.
4. 사이드 패널과 메인 콘텐츠가 병치되면 split layout 리사이즈를 기본 검토한다.

### 2.3 Responsive and Mobile
1. 반응형은 허용하되, 모바일 대응 여부는 실제 사용 맥락으로 판단한다.
2. 모바일은 핵심 정보와 핵심 액션 우선으로 재구성한다.
3. 모바일 UI는 데스크톱 축소본이 아니라 행동 중심 재설계여야 한다.

### 2.4 Lists and Tables
1. 목록은 조회가 아니라 판단과 실행의 출발점이어야 한다.
2. 검색, 필터, 정렬을 기본 제공한다.
3. 상태, 우선순위, 담당자, 예외 여부를 목록 단계에서 빠르게 스캔 가능해야 한다.
4. 행 선택, 상세 진입, 일괄 작업, 퀵 액션의 의미 충돌을 막는다.

### 2.5 Detail Views
1. 상세 화면은 조회, 판단, 실행, 협업, 이력이 한 문맥으로 이어져야 한다.
2. 상태, 핵심 메타데이터, 핵심 액션은 즉시 접근 가능해야 한다.
3. 댓글, 노트, 승인 정보, 첨부, 이력은 필요 시 즉시 열람 가능해야 한다.

### 2.6 Forms
1. 폼은 신규 작성보다 수정, 검토, 보완, 재제출 중심으로 설계한다.
2. 긴 폼은 섹션 구조를 반드시 가진다.
3. 필수 입력, 자동 계산, 시스템 생성, 권한 제한 필드를 구분한다.
4. 검증은 제출 시점뿐 아니라 입력 중에도 제공한다.
5. 수정 불가 필드는 비활성화만 하지 말고 사유를 제공한다.

### 2.7 Workflow and Actions
1. 모든 핵심 객체는 현재 상태와 다음 가능한 액션을 함께 보여준다.
2. 화면의 Primary Action은 하나만 명확해야 한다.
3. 저장, 제출, 승인, 반려, 완료, 삭제 등 결과가 다른 액션은 명확히 구분한다.
4. 고영향 액션은 영향 범위와 결과를 사전에 이해 가능해야 한다.

### 2.8 Feedback and Status
1. 모든 사용자 액션에 즉시 이해 가능한 피드백을 제공한다.
2. 로딩, 저장 중, 동기화 중, 승인 처리 중, 실패, 부분 성공을 사용자 관점으로 표현한다.
3. 실패 메시지는 원인만이 아니라 다음 행동까지 안내한다.
4. 비동기 처리 결과는 나중에 다시 확인 가능해야 한다.

### 2.9 Permissions and Language
1. 권한 차이는 예측 가능하고 일관된 방식으로 드러나야 한다.
2. 권한 부족은 시스템 오류처럼 보이면 안 된다.
3. 관리자 기능은 일반 사용자 핵심 흐름과 분리한다.
4. 버튼명은 결과가 드러나는 동사를 사용한다.
5. 상태명, 필드명, 액션명은 도메인 용어를 일관되게 유지한다.

## 3) Should Rules (Strong Recommendation)

### 3.1 Context Preservation Patterns
1. 전체 페이지 전환 전 패널, 드로어, 인라인 확장 패턴을 우선 검토한다.
2. 목록 복귀 시 필터, 정렬, 스크롤 위치를 유지한다.
3. 현재 위치는 메뉴, 브레드크럼, 타이틀, 객체명, 상태 중 2개 이상으로 표시한다.

### 3.2 Desktop and Mobile Optimization
1. 고밀도 데스크톱 화면에서는 정보 밀도 조절 수단(리사이즈)을 제공한다.
2. 모바일 우선 대상은 확인, 승인, 코멘트, 알림 대응, 상태 확인, 간단 입력으로 본다.
3. 대량 입력, 복합 비교, 다중 컬럼 분석, 관리자 설정은 데스크톱 우선으로 둔다.
4. 모바일 저빈도 액션은 더보기 메뉴로 정리한다.

### 3.3 Execution Efficiency
1. 반복 업무에서는 클릭 수, 이동 수, 입력 수를 줄이는 방향을 우선 검토한다.
2. 일괄 작업, 최근값 재사용, 기본값 자동화, 빠른 액션, 단축 진입을 검토한다.
3. 사용자의 기억보다 시스템 추천이 앞서도록 설계한다.

### 3.4 Writing and Feedback Quality
1. 실패 메시지는 원인, 영향, 다음 행동 순으로 작성한다.
2. 부분 실패와 전체 실패를 명확히 구분한다.
3. 모바일에서는 긴 설명보다 핵심 문장과 우선 행동을 먼저 보여준다.

## 4) Avoid Rules (Anti-patterns)

### 4.1 Layout Anti-patterns
1. 데스크톱 복합 화면을 모바일에 그대로 축소해 넣지 않는다.
2. 모든 화면을 동일 수준으로 모바일 최적화하려 하지 않는다.
3. 맥락이 중요한 작업을 불필요한 전체 페이지 전환으로 끊지 않는다.
4. 탐색, 작업, 보조 정보 역할이 섞인 구조를 피한다.

### 4.2 List/Form Anti-patterns
1. 다중 컬럼 테이블을 모바일에서 그대로 유지하지 않는다.
2. 행 클릭, 체크박스, 퀵 액션 의미가 충돌하는 설계를 피한다.
3. 긴 폼을 구조 없이 단일 스크롤로만 제공하지 않는다.
4. 제출 후에만 오류를 알게 되는 설계를 피한다.
5. 수정 불가 필드를 이유 없이 비활성화만 하지 않는다.

### 4.3 Action/Feedback Anti-patterns
1. 상태 라벨만 보여주고 다음 행동이 보이지 않는 설계를 피한다.
2. 한 화면에 여러 Primary Action이 경쟁하지 않게 한다.
3. 고영향 액션을 일반 액션처럼 가볍게 노출하지 않는다.
4. 불가능한 액션을 이유 설명 없이 막아두지 않는다.
5. 실패 메시지를 기술 오류 문장만으로 끝내지 않는다.

### 4.4 Collaboration/Permission Anti-patterns
1. 업무 객체와 협업 기능을 완전히 분리된 공간으로 떼어놓지 않는다.
2. 상태 전이와 결정 맥락이 남지 않는 구조를 피한다.
3. 비동기 처리 결과가 조용히 사라지지 않게 한다.
4. 권한 부족을 시스템 장애처럼 보이게 하지 않는다.
5. 동일 개념에 여러 용어를 혼용하지 않는다.

## 5) Screen Review Checklist
리뷰 시 아래 질문에 모두 답할 수 있어야 한다.

1. 이 화면의 핵심 사용자 목표는 명확한가
2. 현재 대상, 상태, 다음 행동이 동시에 보이는가
3. 맥락을 유지한 채 작업을 이어갈 수 있는가
4. 모바일 대응 필요성을 실제 사용 맥락으로 설명 가능한가
5. 목록에서 판단과 실행이 가능한가
6. 상세에서 협업과 이력이 연결되는가
7. 권한과 상태에 따른 행동 범위가 명확한가
8. 실패 시 사용자가 다음 행동을 이해할 수 있는가

## 6) Fast Acceptance Criteria
새 화면 또는 개편 화면은 아래를 만족하면 통과로 본다.

1. Primary Action이 하나로 명확하다.
2. 상태별 가능한 행동이 UI에서 즉시 드러난다.
3. 목록에서 핵심 판단이 가능하고 상세 이동이 최소화된다.
4. 상세에서 실행, 협업, 이력 확인이 한 흐름으로 이어진다.
5. 실패와 부분 실패에 대해 다음 행동 안내가 존재한다.
6. 권한 차이가 버그가 아닌 정책으로 인지된다.
7. 반복 사용 시 입력/클릭 부담이 감소한다.

## 7) Reference Message Template
실패 메시지 기본 템플릿:
원인 + 영향 + 다음 행동

예시:
일부 항목 저장 실패. 3건은 권한 부족으로 반영되지 않았습니다. 실패 항목만 검토 후 다시 저장하세요.

## 8) Task Management UI Baseline (Jira/ClickUp Tone)
태스크 매니지먼트 화면은 아래 기준을 기본값으로 사용한다.

### 8.1 View System
1. 태스크의 주요 뷰(List, Board, Backlog, Hierarchy, Decision Graph)는 하나의 탭 시스템에서 전환한다.
2. 동일 업무 맥락에서 가능한 뷰 전환은 사이드 메뉴 분리보다 탭 전환을 우선한다.
3. 활성 탭은 색 대비와 형태 대비로 즉시 구분 가능해야 한다.

### 8.2 Select Control
1. Select의 라벨은 필드 외부가 아니라 필드 내부 포함형으로 기본 제공한다.
2. 라벨은 항상 보이는 compact pill 형식으로 유지한다.
3. 값 영역과 라벨 영역은 시각적으로 분리되어 스캔이 빨라야 한다.
4. hover/focus 상태는 경계선, 포커스 링, 배경 변화로 명확히 구분한다.

### 8.3 Dense List Interaction
1. 리스트 행 hover 시 배경만 바꾸지 말고 상태선 또는 우선순위 신호를 함께 강조한다.
2. 목록 단계에서 상태, 우선순위, 담당자, 신호(노트/스레드)를 한눈에 비교 가능해야 한다.
3. 탭 카운트, 상태 배지, 칩은 과장된 장식보다 스캔 속도와 밀도를 우선한다.
