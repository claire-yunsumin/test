# Objective 1 — Unmet Needs 해소 + Retention (MVP 1)

> **비개발부서와의 의사결정에서 결정 대상이 구조로 드러나고 논의-결정 흐름이 원활해져 Unmet Needs가 해소되며, 이 구조가 Hook Model 루프로 동작하여 사용자가 자발적으로 돌아오는 상태**

달성 시 판단:

- Unmet Needs: "결정 대상이 명확히 보이고, 논의가 그 대상에 귀속되어 흐르는가?" -> Yes/No
- Retention: "사용자가 강제 없이 자발적으로 시스템에 돌아와 루프를 반복하는가?" -> Yes/No

## WHAT — 전략 과제

| # | 전략 과제 | 해소하는 Unmet Needs | Retention (Hook 단계) | DG 근거 |
| --- | --- | --- | --- | --- |
| W1 | 결정 대상의 구조화 | 결정 대상의 불명확 | Investment -> Variable Reward | DG-01 (hierarchy_meaning), DG-07 |
| W2 | 논의-결정 흐름의 귀속 | 논의-결정 흐름 단절 | Trigger -> Action | DG-04, DG-06, DG-14, DG-15 |

## HOW — 접근 방법

| WHAT | Unmet Needs를 이렇게 해소한다 | 이것이 Hook 루프가 되는 이유 |
| --- | --- | --- |
| W1 결정 대상 구조화 | 전략 계층(Vision->Objective->KR->Task)으로 결정 대상을 구조화. hierarchy_meaning으로 도메인 Template(마케팅전략·제품전략 등)이 Objective에 참여하여 "무엇에 대한 결정인지" 구조에 드러남. 비개발부서도 "제품전략의 이 KR에 대해 논의하자"고 지칭할 수 있는 구체적 대상이 생김 | 결정 대상을 형상화하는 행위 자체가 **Investment**. 비개발부서의 피드백으로 구조가 정교해지면 **Variable Reward** — "내가 만든 구조가 타인의 눈을 통해 새롭게 보이는 경험" |
| W2 논의-결정 귀속 | 논의(Thread)와 논의 과정 문서(Notes: 파일·Rich Text)가 결정 대상(Task Instance)에 귀속. 상태전이 시 Notes를 참조하여 Form Output을 승인·반려하고, 논의 맥락과 결정 결과가 함께 보존됨 | 결정 대상에 변경·논의가 발생하면 분류된 알람(**Trigger**). 인지/논의/결정으로 분류되어 즉시 행동(**Action**) 가능. 행동 결과가 구조에 반영되면 다시 Investment로 |

## KR 1.1 — OKR 구조와 Backlog가 Work Graph에 형상화·정형화되어 멘션으로 논의가 시작되는 상태

**Unmet Needs 해소**: 결정 대상의 불명확 -> Work Graph에 형상화·정형화되어 멘션할 수 있는 대상이 됨

> **Retention (Hook)**: 형상화·정형화하는 행위(Investment) + 멘션·피드백으로 정교해짐(Variable Reward)

세 단계가 순차적으로 작동한다. **형상화 -> 정형화 -> 멘션**이 기본 Loop다.

### 1단계 — 형상화: Work Graph에 자유롭게 표현

논의가 시작되려면 결정 대상이 먼저 Work Graph 위에 존재해야 한다. 머릿속에만 있는 것은 가리킬 수 없고, 가리킬 수 없으면 멘션도 논의도 시작되지 않는다. 사용자가 생각나는 대로 노드를 만들고, 관계를 잇고, 이름을 붙이는 — 이 자유로운 형상화가 먼저 보장되어야 한다.

```text
형상화되지 않은 상태 (현재):
  회의 시작 -> "오늘 뭘 결정하지?" -> 맥락 공유가 팀즈, 구두 등 분산
  비개발부서: "마케팅전략 방향을 논의하고 싶은데"
  개발부서: "그게 어떤 항목을 말하는 건지?" -> 대상이 불일치
  -> 가리킬 대상이 없으므로 멘션 불가. 논의 시작점이 없음

Work Graph에 형상화된 상태:
  사용자가 자유롭게 Playbook의 OKR 구조를 표현
  "S/W 제품 마케팅전략"
    ├── "시장성 검증"
    │     ├── "타깃시장 조사"
    │     └── "고객 인터뷰"
    └── "경쟁환경 파악"

  "S/W 제품전략"
    ├── "Phase 1 — MVP 출시"
    │     ├── "File Block 구현"
    │     └── "Timeline 구현"
    └── "Phase 2 — AI 연동"

  -> 계층 규칙이나 Template을 몰라도 노드를 만들 수 있음
  -> 어떤 도메인이든 자유롭게 표현 가능
```

### 2단계 — 정형화: 구조가 드러나면 Template을 적용

형상화된 노드 중 반복되거나 규칙이 필요한 것에 Template을 적용한다. Template이 적용되면 Form Output·Workflow·검수 기준이 활성화되어 "이 노드에서 무엇을 산출해야 하는지"가 명확해진다. 정형화는 강제가 아니다 — 형상화 상태로 유지해도 유효하고, 구조가 드러나는 시점에 점진적으로 적용한다.

```text
정형화 전 (반정형 노드):
  "마케팅전략" — content만 있음. 산출물 구조 없음
  -> 논의는 가능하지만 "무엇을 산출해야 하는지"는 모호

정형화 후 (Template 적용):
  "마케팅전략" [Objective Template 적용]
    Form Output 활성화:
      M1. 문제정의
      M2. 시장 분석 (ICP, Champion, B-PP-UN, 경쟁환경, White Space)
      M3. 게임의 룰 (핵심 경쟁 기준, 지속 가능성 근거)
      M4. 포지셔닝 (포지셔닝 한 문장, 포지셔닝 맵)
    Workflow 활성화:
      작성중 -> 검토대기 -> 승인 (On Exit: M2~M4 완성 여부 검증)
    검수 기준 활성화:
      "ICP 정의가 충분한가?", "White Space 근거가 있는가?"

  "Phase 1 — MVP 출시" [Phase Template 적용]
    Form Output 활성화:
      Release Spec · KR-1.1 출시 · KR-1.2 품질
    Workflow 활성화:
      설계 -> 구현 -> 검증 -> 출시

  -> 정형화된 노드는 "무엇을 해야 하는지" 구조적으로 명확
  -> 논의 대상이 구체적: @마케팅전략.M2 "시장분석 결과를 논의하자"
```

### 3단계 — 멘션: 형상화·정형화된 노드를 가리켜 논의 시작

형상화·정형화된 노드가 있어야 "이것에 대해 논의하자"고 멘션할 수 있다. 멘션이 시작되어야 논의가 흐르고, 논의가 흘러야 결정이 귀속된다.

```text
  비개발부서: @red "고객 인터뷰 결과가 나왔는데, 방향을 재검토하자"
  개발부서: @thomas "이 범위면 기술적으로 2주 더 필요"
  -> 멘션 = 논의의 시작점. 대상이 명확하므로 맥락 공유 비용 최소

기본 Loop:
  형상화 (노드 생성·관계 연결)
    -> 정형화 (Template 적용 — 필요한 노드만, 점진적)
      -> 멘션 (형상화·정형화된 노드를 가리킴)
        -> 피드백 (논의·수정·정교화)
          -> 다시 형상화·정형화 ...
```

### Unmet Needs 해소 기준

| 세부 기준 | Key Index | 유형 |
| --- | --- | --- |
| Playbook의 OKR 구조(Objective->KR->Task)를 Work Graph에 자유롭게 형상화한 상태 | 형상화된 노드 수 · 관계 연결 수 | 정량 |
| 형상화된 노드 중 반복·규칙이 필요한 것에 Template을 적용하여 정형화한 상태 | Template 적용 노드 수 · Form Output 활성화 수 | 정량 |
| 형상화·정형화된 노드를 멘션하여 논의가 시작된 상태 | 노드 멘션 발생 수 · 멘션 기반 Thread 수 | 정량 |
| 참여자(비개발·개발)가 동일한 노드를 지칭하여 논의하는 상태 | 크로스펑셔널 Thread 발생률 | 정량 |

### Retention 기준 (Hook)

| 세부 기준 | Key Index | 유형 |
| --- | --- | --- |
| 형상화·정형화(Investment)가 일회성이 아니라 지속적으로 일어나는 상태 | 주간 노드 생성·수정·Template 적용 빈도 | 정량 |
| 멘션·피드백(Variable Reward)으로 구조가 정교해지는 경험이 반복되는 상태 | 피드백 후 노드 수정 발생률 | 정량 |

## KR 1.2 — Notes 기반 논의, 합의/승인 Gate를 제공해 알람이 스트레스 없이 Action Flow로 흐르도록 하여 Loop가 반복되는 상태

**Unmet Needs 해소**: 논의-결정 흐름 단절 -> Notes·Thread·합의/승인이 결정 대상에 귀속되고, 발생한 이벤트가 분류되어 전달

> **Retention (Hook)**: 분류된 알람(Trigger) + Notes 기반 논의 + 합의/승인 Gate 통과(Action) -> 행동 결과가 다시 Investment로

KR 1.1에서 형상화·정형화·멘션으로 기본 Loop가 돌아가면, 그 과정에서 논의·결정·공유 이벤트가 발생한다. 이 이벤트가 사용자에게 **분류되어** 도착해야 하고, 도착한 알람에 대해 **Notes로 논의하고 합의/승인 Gate를 통과**해야 결정이 구조에 귀속된다. 이 세 가지가 갖춰져야 Loop가 완성된다.

```text
Loop가 불완전한 상태:
  알람은 분류되어 도착하지만 —
  ① Notes 없이 구두로 논의 -> 논의 맥락이 휘발
  ② 합의/승인 Gate 없이 상태전이 -> "누가 승인한 거야?" 추적 불가
  -> Action의 결과가 구조에 귀속되지 않음 -> Loop가 허공에서 끊김

Loop가 완성된 상태:
  알람 분류 (Trigger)
    -> Notes 기반 논의 (Action — 맥락이 구조에 남음)
    -> 합의/승인 Gate 통과 (Action — 결정이 구조에 귀속)
    -> 상태전이 + Timeline 기록
    -> 다시 Investment (형상화·정형화)로
```

### ① 알람 분류 — 사용자 기대 행동 × 긴급도 기준

결정과 논의는 즉시 사용자 액션을 요구하며, 인지는 상황 파악 후 후속 판단을, 결과는 필요 시 참조하는 구조다. **결정 큐 내에서 승인·보완요청·반려가 하나의 라이프사이클로 흐르므로 큐 간 파생 없이 결정의 전체 맥락이 보존된다.** 세부 트리거·상태 모델·UI 흐름은 System Spec에서 정의한다.

#### Alarm Components — 4가지 액션 유형 (사용자 기대 행동 × 긴급도 기준)

| 유형 | 긴급도 | 사용자 기대 행동 | 유입 항목 |
| --- | --- | --- | --- |
| **결정** | 즉시 | 판단을 내린다 | 승인·반려·보완요청 등 결정 관련 전체 라이프사이클. 결정에 수반되는 커뮤니케이션(보완 사유·반려 사유)도 이 큐에 통합 |
| **논의** | 즉시 | 의견을 나눈다 | Thread 멘션·질문·의견 요청. 상대방의 응답이 필요한 협업 커뮤니케이션 |
| **인지** | 알고 있어야 함 | 변경을 파악한다 | 할당·재오픈·기한 초과·상위 Task 변경 전파·Link 영향 전파. 사용자가 상황을 인지하고 후속 판단을 준비 |
| **결과** | 필요 시 참조 | 결과를 확인한다 | 승인 결과·완료 통보·자동화 결과. 입력 없는 인지 전용(읽고 끝) |

```text
알람이 분류되지 않은 상태 (도입 전):
  이메일 50건 · Teams 알림 30건 · 구두 요청 5건
  -> "뭐부터 해야 하지?" -> 알람 끔 -> 액션 누락 -> 루프 멈춤

알람이 유형별로 분류된 상태:
  결정 (즉시 — 판단을 내린다)
    "마케팅전략 M2~M4 승인 요청" -> 승인/보완요청/반려
    -> 보완요청 시 보완 사유도 결정 큐 안에서 흐름
    -> 보완 완료 -> 재승인 요청 -> 최종 승인/반려
    -> 결정의 전체 라이프사이클이 하나의 큐에서 완결

  논의 (즉시 — 의견을 나눈다)
    @시장성검증 "고객 인터뷰 결과를 보고 방향을 재검토하자"
    -> 논의 참여 -> Thread에서 의견 교환

  인지 (알고 있어야 함 — 변경을 파악한다)
    "Phase 1 Release Spec 작성이 할당되었습니다"
    "상위 Objective '마케팅전략'이 변경되었습니다"
    -> 상황 파악 후 후속 판단 준비

  결과 (필요 시 참조 — 결과를 확인한다)
    "Phase 1 Release Spec 승인 완료" -> 읽고 끝
    -> 별도 액션 불필요

  -> 사용자는 "나에게 온 것"만 보면 됨
  -> 긴급도와 기대 행동이 명확하므로 "결정부터, 논의 다음, 인지는 여유 있을 때"
  -> 스트레스 없이 처리 -> 알람을 끄지 않음 -> 루프가 지속됨
```

### ② Notes — 논의 맥락이 결정 대상에 귀속되는 구조

알람을 받고 행동하려면 **논의할 자료**가 결정 대상에 붙어 있어야 한다. Notes(파일·Rich Text)가 결정 대상인 Task Instance에 귀속되면, 논의 참여자가 같은 자료를 보고 같은 맥락에서 논의할 수 있다. Notes 없이 논의하면 맥락이 이메일·메신저로 흩어지고, Loop가 구조에 귀속되지 않는다.

```text
결정 대상: Objective "마케팅전략" > KR "시장성 검증"

Notes (결정 대상에 귀속 — Output을 논의하는 과정 문서)
  ├── 시장분석.pdf (비개발부서가 첨부한 파일)
  ├── 기술검토 Rich Text (개발부서가 작성)
  └── 고객인터뷰_요약.md (GTM KR-1 현장검증 결과)

Form Output (결정 대상에 귀속 — 구조화된 산출물)
  ├── 값 Block: 핵심 판단 근거
  └── 참조 Block: 관련 Task 데이터

Thread (결정 대상에 귀속)
  비개발부서: "시장 규모가 예상보다 작은데 방향을 재검토해야 할까?"
  개발부서: "기술적으로는 가능하지만 일정이 2주 더 필요"
  -> 논의가 결정 대상 안에서 일어남
  -> Notes를 참조하며 Thread에서 의견 교환 (노트 연결은 #으로 참조)
```

### ③ 합의/승인 Gate — 결정이 구조에 귀속되는 관문

논의가 끝나면 **합의/승인 Gate를 통과해야** 상태전이가 일어난다. Gate가 없으면 "누가 언제 승인했는지" 모른 채 상태가 바뀌고, 결정의 책임이 불명확해진다. Gate를 통과하면 **누가 · 언제 · 어떤 Notes를 보고 · 왜 승인/반려했는지**가 Timeline에 자동 기록되어 결정이 구조에 귀속된다.

```text
합의/승인 Gate 흐름 (Playbook 예시):

  KR "시장성 검증" — Workflow 상태전이
    작성중 -> 검토대기
      On Exit: M2(시장분석) Block 완성 여부 검증
    검토대기 -> [합의/승인 Gate]
      승인자: 비개발부서 담당자, 개발부서 리드
      판단 근거: Notes(시장분석.pdf, 고객인터뷰_요약.md)
      -> 승인  -> 상태전이: 검토대기 -> 승인됨
      -> 보완요청 -> 보완 사유 기록 -> 결정 큐 안에서 라이프사이클 지속
      -> 반려  -> 반려 사유 기록 -> 결정 큐 안에서 종결

  Timeline 자동 기록:
    [2026-04-23 14:30] 검토대기 -> 승인됨
    승인자: @dennis, @claire
    참조 Notes: 시장분석.pdf, 고객인터뷰_요약.md
    사유: "ICP 검증 완료. White Space 근거 충분"
    -> 사후 추적: "왜 이렇게 결정했지?" -> Timeline에서 전체 맥락 복원

합의/승인 Gate가 Loop를 완성하는 이유:
  형상화·정형화 (KR 1.1) -> 멘션으로 논의 시작
    -> 알람 분류로 도착 (Trigger)
    -> Notes 기반 논의 (Action — 맥락 귀속)
    -> 합의/승인 Gate 통과 (Action — 결정 귀속)
    -> Timeline 기록 (결정의 구조적 증거)
    -> 다시 형상화·정형화로 (Investment)
```

### Unmet Needs 해소 기준

| 세부 기준 | Key Index | 유형 |
| --- | --- | --- |
| 논의 자료(Notes)가 결정 대상에 귀속되어 비개발·개발부서가 같은 자료를 보고 논의하는 상태 | 의사결정 시 Notes 활용률 | 정량 |
| 논의 내용(Thread)이 결정 대상에 귀속되어 맥락이 보존되는 상태 | 결정 대상 귀속 Thread 비율 | 정량 |
| 합의/승인 Gate를 통과하여 결정 결과(승인·반려·보완)가 Timeline에 기록되고 "어떤 Notes를 보고 왜 결정했는지" 추적 가능한 상태 | Gate 통과 후 Timeline 추적 가능 (Pass/Fail) | 비정량 |
| 전략(Objective)->로드맵(KR)->할일(Task)가 하나의 계층으로 관리되어 결정의 영향 범위가 보이는 상태 | 전략-실행 계층 연결 (Pass/Fail) | 비정량 |

### Retention 기준 (Hook)

| 세부 기준 | Key Index | 유형 |
| --- | --- | --- |
| Alarm 분류(결정/논의/인지/결과)를 통해 "무엇을 해야 하는지" 즉시 파악하여 행동(Action)으로 진입하는 상태 | Alarm->Action 전환율 | 정량 |
| 알람을 끄지 않고 처리하는 상태 (스트레스 없는 Action Flow) | Alarm 무시율 (낮을수록 좋음) | 정량 |
| Notes 기반 논의 -> 합의/승인 Gate 통과 -> Timeline 기록의 흐름이 완결되어 결정이 구조에 귀속되는 상태 | Gate 통과율 · 결정 귀속 완결률 | 정량 |
| 행동(논의·결정) 결과가 구조에 반영되어 다시 형상화·정형화(Investment)로 이어지는 상태 | Action 후 구조 갱신 발생률 | 정량 |
| 강제 없이 사용자가 자발적으로 시스템에 돌아와 루프를 반복하는 상태 | 주간 자발적 재방문율 | 정량 |

# Objective 2 — AI 활용 + 변경 인지 (MVP 2)

> **Task 산출물에 AI를 활용할 수 없어 불편한 문제가 해소되고, 관계 규칙에 의한 변경이 관련자에게 적절히 인지되는 상태** (기술 불확실성을 해소하여 방법론을 Template으로 표현하기 위한 방안이 설계된 상태)

달성 시 판단: "AI가 Template에 선언된 방법론을 읽고 산출물 품질을 높이는가? 상위 변경이 하위 실행자에게 적절히 전파되는가?" -> Yes/No

**O1과의 관계**: O1에서 Unmet Needs를 해소하며 만들어진 결정 대상(계층 노드)·논의 자료(Notes)·논의 흐름(Thread)·결정 기록(상태전이) 위에 AI와 변경 전파가 얹어진다. 결정 대상이 구조화되지 않은 상태에서 AI를 투입해도 "무엇에 대해 도와야 하는지" AI도 모르고, 변경이 전파될 경로(계층)도 없다.

## WHAT — 전략 과제

| # | 전략 과제 | 해소하는 Unmet Needs | DG 근거 |
| --- | --- | --- | --- |
| W3 | Task 산출물에 대한 AI 활용 구조 확립 | AI 활용 불가 | DG-02, DG-06, DG-08 |
| W4 | 관계 규칙에 의한 변경의 인지 구조 확립 | 변경 전파 부재 | DG-04, DG-05, DG-15 |

## HOW — 접근 방법

| WHAT | HOW |
| --- | --- |
| W3 AI 활용 | Notes(논의 문서)를 AI가 생성·보완하고, Form Output(산출물)을 검증·가이드. Task 내에서 맥락 기반 질의 가능. Template의 Output 목적·검수 기준·AI Context(product·strategy·constraints)가 AI에게 방법론 지식을 전달 |
| W4 변경 인지 | 관계 규칙(hierarchy·related)을 통해 상위 노드 변경 시 영향 범위를 자동 산출. 즉시 대응이 아닌 **실행 시점에 변경을 인지**하거나, **변경에 따라 새로 생긴 할 일만 알림**으로 전달하는 Pull 기반 인지 모델 |

## KR 2.1 — AI가 Task 산출물의 생성·검증·가이드를 제공하여 올바른 Output이 나오는 상태

> 기술 불확실성 해소: "Template에 선언된 방법론이 AI에게 전달되어 산출물 품질에 실질적 차이를 만드는가?"

Task 내에서 AI가 세 가지 역할을 수행한다.

### 역할 1. Notes 생성과 Task 내 질의

```text
AI가 하는 것:
  Notes 초안 생성     상위 Objective·관련 Task 데이터를 참조하여
                      논의용 문서(파일 또는 Rich Text) 초안 작성
  Task 내 질의        Task의 맥락(계층 위치·AI Context·Form 데이터)을 바탕으로
                      사용자 질문에 답변. "이 KR의 달성 기준이 적절한가?"

사용자가 하는 것:
  AI 초안 검토 -> 수정 -> 확정 (§0 "최종 확정은 사람이 한다")
```

| 세부 기준 | Key Index | 유형 |
| --- | --- | --- |
| AI가 Task 맥락을 읽고 Notes 초안을 생성하여 사용자 작성 시간이 단축되는 상태 | Notes AI 생성 활용률 | 정량 |
| Task 내 질의에 대해 AI가 계층 맥락 기반으로 의미 있는 답변을 제공하는 상태 | Task 내 AI 질의 유용성 (Pass/Fail) | 비정량 |

### 역할 2. Template 기반 Output 가이드와 검수

```text
Template이 선언하는 것:
  Output 목적         "이 Task의 산출물이 달성해야 하는 것"
  검수 기준            "산출물이 충족해야 하는 품질 기준" (DG-04 inspection_criteria)
  AI Context          도메인 방법론 (product · strategy · constraints)

AI가 하는 것:
  작성 가이드     Template의 Output 목적·AI Context를 읽고
                  "이 Block에는 이런 내용이 필요합니다" 안내
  검수 체크       On Exit 전에 검수 기준 대비 산출물 분석
                  "검수 기준 3개 중 2개 충족. 미충족 항목: ..."
  개선 제안       미충족 항목에 대해 구체적 개선 방향 제시

사용자가 하는 것:
  AI 가이드 참고 -> 산출물 작성 -> AI 검수 결과 확인 -> 보완 -> 확정
```

| 세부 기준 | Key Index | 유형 |
| --- | --- | --- |
| AI가 Template의 Output 목적·검수 기준을 읽고 산출물을 검증하는 상태 | AI 검수 기반 산출물 개선 건수 | 정량 |
| Template에 방법론을 선언하면 AI가 해당 방법론에 맞는 가이드를 제공하는 상태 | Template->AI 가이드 정합성 (Pass/Fail) | 비정량 |

### 기술 불확실성 검증 포인트

| 질문 | 검증 방법 |
| --- | --- |
| Template에 선언된 방법론이 AI에게 충분히 전달되는가? | AI Context 구조로 주입 -> 산출물 품질 비교 (with/without) |
| 검수 기준이 AI가 판단할 수 있는 수준으로 선언 가능한가? | inspection_criteria 다양한 포맷 실험 -> 판정 정확도 측정 |

## KR 2.2 — 관계 규칙에 의한 변경이 관련자에게 적절히 인지되어 변경 누락 없이 실행할 수 있는 상태

> 핵심 원칙: **변경될 때마다 사람이 일일이 파악하는 것은 불가능하다.** 실행할 때 변경을 인지하거나, 변경에 따라 새로 해야 할 일만 아는 것이 좋다.

### 문제 구조

```text
현재:
  상위 Objective 변경 -> 관련 KR·Task 담당자가 모름
  -> 변경을 아는 사람이 구두·메신저로 일일이 전달
  -> 누군가 빠뜨리면 변경이 반영되지 않은 채 실행 계속
  -> 실행 완료 후에야 "상위가 바뀌었는데 왜 예전 기준으로 했어?" 발생
```

### 설계 원칙: Push가 아닌 Pull 기반 변경 인지

변경이 발생할 때마다 즉시 대응을 요구하면 알람 폭주로 이어진다. 두 가지 방식으로 변경을 인지한다.

```text
방식 1. 실행 시점 인지 (Pull)
  사용자가 Task를 열거나 상태전이를 시도할 때
  시스템이 상위·관련 노드의 변경 여부를 확인
  변경이 있으면 인지 마커(🟡)로 표시
  "상위 Objective가 4/18에 변경되었습니다. 확인하시겠습니까?"
  -> 사용자가 실행하려는 시점에 필요한 변경만 인지

방식 2. 새 할 일 알림 (Push, 최소)
  변경에 의해 새로운 Action이 생긴 경우에만 알림
  "상위 KR 변경으로 인해 검토가 필요한 Task가 1건 있습니다"
  -> Alarm Components의 Inbox(유입 게이트)로 유입
  -> 기존 진행 중 Task에 대한 "참고" 수준 변경은 알림하지 않음
```

### 관계 규칙이 변경 전파의 경로다

```text
관계 규칙 (DG-04 Workflow)
  hierarchy    상위-하위 관계 -> 상위 변경 시 하위에 인지 마커
  related      연관 관계 -> 관련 노드 변경 시 인지 마커
  derived_from 파생 원본 -> 원본 변경 시 파생 노드에 인지 마커

전파 범위 산출:
  변경된 노드 -> 관계 규칙으로 연결된 노드 목록 산출
  -> 직접 연결(depth 1)만 기본. 연쇄 전파는 단계적 확장
```

### 하네스엔지니어링 예시

```text
Objective "S/W 제품 마케팅전략" 변경
  ├── KR "시장성 검증" 담당자 -> Task 열 때 🟡 "상위 Objective 변경됨"
  ├── Task "타깃시장 조사" 담당자 -> 상태전이 시도 시 🟡 표시
  └── related Task "경쟁사 분석" 담당자 -> 변경 확인 전까지 🟡 유지

  Objective 변경으로 새 KR이 필요한 경우:
  -> "마케팅전략 변경에 따라 새 KR 정의가 필요합니다" Inbox 알림
  -> 기존 KR·Task에 대해서는 실행 시점에 인지 (Push 하지 않음)
```

| 세부 기준 | Key Index | 유형 |
| --- | --- | --- |
| 상위 노드 변경 시 하위·관련 노드에 인지 마커가 자동 부여되는 상태 | 변경 전파 인지 마커 동작 (Pass/Fail) | 비정량 |
| 사용자가 Task 실행 시점에 상위 변경 사항을 확인하고 반영할 수 있는 상태 | 실행 시점 변경 인지율 | 정량 |
| 변경에 의해 새로 생긴 할 일만 Inbox 알림으로 유입되는 상태 (알람 폭주 없음) | 변경 기반 Inbox 유입 정확도 | 정량 |
