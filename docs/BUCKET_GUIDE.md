# Bucket Guide

버킷은 태스크의 계층, 상태, 워크플로우와 별개로 실행 묶음을 만드는 운영 축입니다. 현재 구현에서는 `Task.bucketId`와 `/api/buckets` API로 관리하며, 리스트/보드/백로그/결정 그래프 뷰에서 작업 맥락을 보조합니다.

## 역할

- 계층: Objective -> KR -> Task처럼 결정 대상의 의미 구조를 표현합니다.
- 워크플로우 상태: Backlog, Active, Review, Done처럼 진행 단계를 표현합니다.
- 버킷: "우선 처리", "버그 수정", "아이디어", "대기"처럼 같은 리스트 안에서 실행 큐를 나눕니다.

버킷은 상태값을 대체하지 않습니다. 상태 변경은 결정/진행 이력이고, 버킷 변경은 운영상 묶음 조정입니다.

## API

- `GET /api/buckets`: 버킷 목록 조회
- `POST /api/buckets`: 버킷 생성
- `PATCH /api/buckets/:bucketId`: 이름, 색상, 순서 수정
- `DELETE /api/buckets/:bucketId`: 버킷 삭제
- `PATCH /api/tasks/:taskId`: 태스크의 `bucketId` 변경

버킷 삭제 시 연결된 태스크는 버킷 없음 상태로 남아야 합니다. 삭제가 태스크 삭제나 상태 전이를 유발해서는 안 됩니다.

## UI 운영

### 초기 세팅

- 태스크 화면에서 버킷 관리 패널을 열어 기본 버킷을 만듭니다.
- 추천 기본값: `우선 처리`, `버그 수정`, `아이디어`, `대기`
- 버킷 순서는 팀이 실제로 훑는 순서에 맞춥니다.

### 일일 운영

- 데일리에서는 버킷별로 병목과 우선순위를 확인합니다.
- `우선 처리`는 오늘 바로 처리할 실행 큐로 사용합니다.
- 버킷 없음 태스크는 주기적으로 정리합니다.

### 뷰별 표시

- 리스트/백로그: 버킷을 태스크 메타 정보로 표시합니다.
- 보드: 상태/컬럼 흐름과 버킷 정보를 같이 보여 줍니다.
- 결정 그래프: 버킷은 구조적 부모-자식 관계가 아니라 보조 그룹 정보로 표시합니다.

## 주의사항

- 버킷 이름을 워크플로우 상태와 중복시키지 않습니다.
- 버킷 수가 많아지면 탐색 비용이 커지므로 4~7개 수준을 권장합니다.
- 의사결정 단위가 필요하면 버킷이 아니라 계층/템플릿/멘션으로 표현합니다.
