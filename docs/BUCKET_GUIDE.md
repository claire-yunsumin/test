# 버킷 지정 가이드

이 문서는 태스크를 버킷으로 묶어 운영할 때의 실무 시나리오를 정리합니다.

## 언제 버킷을 쓰는가

- 계층(상위/하위)과 별개로 작업 묶음을 빠르게 보고 싶을 때
- 같은 리스트 안에서 실행 성격(예: 우선 처리, 버그 수정, 아이디어)을 나눌 때
- 스프린트 중 담당자가 "지금 처리 묶음"을 공유해야 할 때

## 권장 운영 시나리오

## 1) 초기 세팅

- `태스크` 화면에서 보기/그룹 옵션을 `bucket`으로 전환
- 상단 `버킷 관리`에서 기본 버킷 생성
  - 예: `우선 처리`, `버그 수정`, `아이디어`, `대기`
- 버킷 순서를 팀 합의 기준으로 정렬

## 2) 태스크 배치

- 버킷 그룹 모드에서는 행의 버킷 셀렉트로 직접 배치
- 필요 시 드래그앤드롭으로 버킷 간 이동
- 버킷 없음은 임시 상태로 보고, 주기적으로 정리

## 3) 일일 운영

- 데일리에서는 버킷 단위로 진행 현황 점검
- `우선 처리` 버킷을 오늘의 실행 큐로 사용
- 병목이 생기면 버킷 이동으로 우선순위 재조정

## 4) 회고/정리

- 완료 태스크가 누적된 버킷은 정리하거나 이름 재정의
- 더 이상 쓰지 않는 버킷 삭제
  - 삭제 시 연결 태스크는 `버킷 없음`으로 이동

## UI 원칙 (현재 반영)

- 일반 리스트/백로그에서는 버킷을 읽기값으로 표시 (노이즈 최소화)
- 버킷 그룹 모드에서만 버킷 셀렉트를 노출 (변경 작업 집중)

## 빠른 체크리스트

- 버킷 없음 태스크가 과도하게 쌓이지 않는가?
- 버킷 이름이 "상태"와 중복되지 않는가? (상태/버킷 역할 분리)
- 버킷 수가 너무 많아 탐색 비용이 커지지 않는가?
