# O1 최종 달성(100%) 우선순위 실행 계획

## 문서 성격

| 항목 | 내용 |
| --- | --- |
| 문서 타입 | Execution Plan (실행 계획) |
| 분류 기준 | Release Spec과 별도 운영 문서 |
| 목적 | Objective 1 달성 갭을 우선순위 기반으로 닫는 실행 계획 |

## 목표

- O1의 남은 부분달성 항목 3개를 닫아 `Unmet Needs = Yes`, `Retention = Yes`를 운영 지표로 자동 판정 가능한 상태로 만든다.

## 현재 갭 요약

- 알람 4분류는 타입/탭은 있으나 운영 완결(특히 `ack` UX, 이벤트-컴포넌트 고정 매핑) 미흡
- 합의/승인은 전이/정책 골격은 있으나 단계별 UX(요청/승인/반려/보완요청)와 진행 가시성이 약함
- Retention API/화면은 있으나 O1 판정 핵심 지표(`Alarm->Action`, 결정 귀속 완결률, 주간 재방문 엄밀성)가 부족

## 우선순위 실행 항목

### P1. 알람 4분류 운영 완결

| 구분 | 내용 |
| --- | --- |
| 목표 | DECISION/DISCUSSION/AWARENESS/RESULT 분류가 이벤트-처리행동-UI까지 일관되게 닫히도록 구현 |
| 변경 포인트 | 이벤트->Inbox 컴포넌트 매핑 보강, `ack` 상태의 수신/발신 UX 노출, 공용 타입 정합성 업데이트 |
| 완료 기준 | 주요 이벤트 명시 매핑, read/ack/remind/read-all 전부 UI 노출, 홈/Inbox/배지 수치 정합 |

### P2. 합의/승인 라이프사이클 UX 완결

| 구분 | 내용 |
| --- | --- |
| 목표 | 요청 -> 승인/반려/보완요청 -> 재검토 흐름이 Task 상세에서 명확히 보이고 추적 가능 |
| 변경 포인트 | 전이 이벤트 타입 세분화, 타임라인 기록 강화, 승인 단계/남은 승인자/다음 액션 가시화 |
| 완료 기준 | 승인요청 건이 최종 상태로 귀결되고 타임라인에 이유/근거 기록, 승인 단계와 남은 주체를 한 화면에서 확인 가능 |

### P3. O1 Retention 자동 판정 고도화

| 구분 | 내용 |
| --- | --- |
| 목표 | O1 판정을 단순 임계치가 아닌 루프 완결 지표로 자동화 |
| 추가 지표 | `Alarm->Action 전환율`, `결정 귀속 완결률`, `주간 자발 재방문율(7일 윈도우)` |
| 변경 포인트 | 집계 로직/응답 메타(`computedAt`, `dataStatus`) 확장, Analytics 타입 확장, 분석 화면 자동판정 로직 교체 |
| 완료 기준 | 분석 화면에서 O1 최종 판정이 자동 계산되고, 지표 카드가 Trigger->Action->Investment 루프와 1:1 대응 |

## 실행 순서와 이유

- **P1 -> P2 -> P3**
  - P1: 이벤트 분류/행동 데이터의 정합성 확보
  - P2: 결정 라이프사이클 완결 신호 확보
  - P3: 신뢰도 높은 자동 판정 완성

## 검증 전략

| 영역 | 검증 항목 |
| --- | --- |
| API | Inbox 컴포넌트 매핑, ack/read/remind/read-all, split transition/approval APIs, 승인 요청 중복 409, TaskDetailDto 런타임 필드 |
| UI | Inbox 4분류 처리행동, Task 승인단계 가시성, Analytics O1 판정 카드 |
| 판정 | O1 `Unmet Needs/Retention` Yes/No가 이벤트 로그 변화에 따라 재현 가능한지 확인 |
