# Business Rules

이 문서는 현재 제품에서 반드시 지켜야 하는 비즈니스 규칙입니다. 기준은 최신 코드 구현이며, 프론트의 표시 제어가 아니라 서버 검증을 최종 신뢰 경계로 둡니다.

## 1. 사용자와 권한

### 역할 계층

현재 역할은 다음 순서로 비교합니다.

`MEMBER < OWNER < ADMIN < SUPER_ADMIN`

- `MEMBER`: 본인이 관련된 태스크와 상위 맥락을 조회하고, 허용된 협업 행동을 수행합니다.
- `OWNER`: 소유한 작업 단위의 편집 권한을 가집니다.
- `ADMIN`: 워크스페이스 운영, 정책, 멤버, 전체 데이터 관리 권한을 가집니다.
- `SUPER_ADMIN`: 데모/운영 최상위 관리자입니다.

권한 판단은 서버에서 수행합니다. 프론트의 버튼 숨김, 비활성화, 안내 문구는 사용성 보조 장치입니다.

## 2. 가시성

관리자(`ADMIN`, `SUPER_ADMIN`)는 전체 태스크를 볼 수 있습니다.

일반 사용자는 다음 조건에 해당하는 태스크를 볼 수 있습니다.

- 본인이 `ownerId`인 태스크
- 본인이 `assigneeIds`에 포함된 태스크
- 본인이 `watcherIds`에 포함된 태스크
- 위 태스크의 상위 parent chain

노트 참조, 태스크 멘션, 폼 필드 멘션, Inbox/타임라인 노출은 이 가시성 계산 결과를 기준으로 검증합니다.

## 3. 태스크 구조

태스크는 자유 형상화와 템플릿 정형화를 모두 지원합니다.

- `FREEFORM`: 템플릿 없이 Work Graph에 먼저 올려 둔 반정형 결정 대상
- `TEMPLATED`: 템플릿이 적용되어 Form Output, 검수 기준, 워크플로우가 활성화된 결정 대상

태스크는 `unitId`, `folderId`, `listId` 맥락을 가지며, 생성/수정 시 조합 무결성을 검증합니다.

- `listId`는 해당 `unitId`에 속해야 합니다.
- `folderId`는 해당 `unitId`에 속해야 합니다.
- `folderId`와 `listId`는 같은 컨텍스트여야 합니다.
- 조합이 맞지 않으면 `FOLDER_LIST_MISMATCH`로 거부합니다.

## 4. 뷰와 운영 축

현재 태스크 화면의 주요 뷰는 다음과 같습니다.

- 리스트: 밀도 높은 태스크 조작과 필터링
- 보드: 상태 컬럼 기반 진행 흐름
- 백로그: 우선순위 큐와 계획 전 작업
- 결정 그래프: 계층, 템플릿, 멘션, 결정 참조 관계

## 5. 템플릿과 Form Output

템플릿은 다음을 소유합니다.

- `formDefinition`
- `inspectionCriteria`
- `workflow`
- 선택적 `workflowSchema`

템플릿 적용 시 태스크는 `TEMPLATED`가 되고, `formValues`는 템플릿 필드 키 기준으로 초기화됩니다. 이후 Form Output UI는 임의 key/value가 아니라 템플릿 필드 정의를 기준으로 렌더링합니다.

## 6. 노트, 댓글, 멘션

댓글은 `mentions` 배열을 가질 수 있습니다.

- `MEMBER`: 멤버 멘션
- `TASK`: 태스크 멘션
- `FORM_FIELD`: 특정 태스크의 템플릿 필드 멘션
- `NOTE`: 노트 참조

작성 UX는 `@`와 `#`를 칩 더미처럼 쌓는 방식이 아니라, 커맨드 검색 메뉴로 후보를 찾고 선택하면 본문 안에 토큰이 삽입되는 방식입니다.

검증 규칙은 다음과 같습니다.

- `referencedNoteIds`는 작성자가 볼 수 있는 노트만 허용합니다.
- `TASK`, `FORM_FIELD`, `NOTE` 멘션은 작성자의 가시 범위 안에서만 허용합니다.
- `FORM_FIELD`는 대상 태스크의 템플릿 필드 키가 존재해야 합니다.
- 실패 시 `INVALID_NOTE_REFERENCE` 또는 `INVALID_MENTION`을 반환합니다.

## 7. 상태 전이와 결정

기본 `TaskState`는 다음 네 가지입니다.

- `DRAFT`
- `IN_PROGRESS`
- `DONE`
- `CANCELED`

세부 워크플로우는 `WorkflowStatus`와 `WorkflowStatusCategory`로 표현합니다. 승인 대기 같은 단계는 `TaskState`가 아니라 워크플로우 상태 카테고리(`PENDING_APPROVAL`)로 관리합니다.

전이 API는 `reason`을 필수로 받습니다. 결정 타입은 `APPROVE`, `REJECT`, `SUPPLEMENT`, `STATE_ONLY`를 사용하며, 결과는 타임라인과 Inbox에 기록됩니다.

## 8. 승인 정책

승인 정책은 관리자 영역에서 관리합니다.

- 글로벌 정책과 유닛별 정책을 지원합니다.
- 정책은 여러 승인 라인을 가질 수 있습니다.
- 각 라인은 승인자, 조건, 최종 승인자 여부를 가집니다.
- 비활성 정책은 새 결정 흐름에 적용하지 않습니다.

승인 정책은 결정 UX를 구조화하기 위한 운영 규칙이며, 단순 상태 변경과 구분해야 합니다.

## 9. Inbox와 알림

서버는 도메인 이벤트를 Inbox 컴포넌트로 분류합니다.

- `DECISION`
- `DISCUSSION`
- `AWARENESS`
- `RESULT`

사용자는 Inbox 항목을 읽음, 확인, 리마인드 처리할 수 있습니다. 알림 설정은 이메일/푸시/데스크톱 채널 및 컴포넌트별 토글을 가집니다.

## 10. 분석과 추적

주요 행동은 engagement event로 기록합니다.

- 노드 생성/수정
- 부모 변경
- 템플릿 적용
- Form 저장
- 댓글 작성
- 멘션 생성
- 노트 수정
- 상태/결정 전이

분석 API는 고정값이 아니라 현재 메모리 데이터와 이벤트 로그를 기반으로 지표를 계산합니다.

## 11. 테스트와 변경 관리

아래 규칙은 자동화 테스트로 보호해야 합니다.

- 인증/인가
- IDOR 방지와 가시성
- unit/folder/list 무결성
- 노트 참조와 멘션 검증
- 템플릿 적용과 Form Output
- 상태 전이와 Inbox 라우팅
- 분석 지표 계산

비즈니스 룰을 바꾸면 API 스펙, 시스템 스펙, 테스트, 프론트 안내 문구를 함께 갱신합니다.
