# Business Rules

이 문서는 현재 제품에서 반드시 지켜야 하는 비즈니스 규칙입니다. 기준은 최신 코드 구현이며, 프론트의 표시 제어가 아니라 서버 검증을 최종 신뢰 경계로 둡니다.

## 1. 사용자와 권한

### 역할 계층

현재 역할은 다음 순서로 비교합니다.

`MEMBER < OWNER < ADMIN < SUPER_ADMIN`

- `MEMBER`: 본인이 관련된 태스크와 상위 맥락을 조회하고, 허용된 협업 행동을 수행합니다.
- `OWNER`: 소유한 작업 단위의 편집 권한을 가집니다.
- `ADMIN`: 워크스페이스 운영, 정책, 멤버, 전체 데이터 관리 권한을 가집니다.
- `SUPER_ADMIN`: 데모/운영 최상위 관리자입니다.

권한 판단은 서버에서 수행합니다. 프론트의 버튼 숨김, 비활성화, 안내 문구는 사용성 보조 장치입니다.

태스크 조회 가능성과 수정 가능성은 분리됩니다.

- 조회 가능: 관리자 전체, 또는 owner/assignee/watcher 관계와 parent chain
- 태스크 필드 수정 가능: 관리자, task owner, task assignee, 해당 unit owner
- Form/description 수정 가능: 관리자, task owner, task assignee
- watcher 또는 parent chain으로만 보이는 사용자는 read-only입니다.

## 2. 가시성

관리자(`ADMIN`, `SUPER_ADMIN`)는 전체 태스크를 볼 수 있습니다.

일반 사용자는 다음 조건에 해당하는 태스크를 볼 수 있습니다.

- 본인이 `ownerId`인 태스크
- 본인이 `assigneeIds`에 포함된 태스크
- 본인이 `watcherIds`에 포함된 태스크
- 위 태스크의 상위 parent chain

노트 참조, 태스크 멘션, 폼 필드 멘션, Inbox/타임라인 노출은 이 가시성 계산 결과를 기준으로 검증합니다.

## 3. 태스크 구조

태스크는 자유 형상화와 템플릿 정형화를 모두 지원합니다.

- `FREEFORM`: 템플릿 없이 Work Graph에 먼저 올려 둔 반정형 결정 대상
- `TEMPLATED`: 템플릿이 적용되어 Form Output, 검수 기준, 워크플로우가 활성화된 결정 대상

태스크는 `unitId`, `folderId`, `listId` 맥락을 가지며, 생성/수정 시 조합 무결성을 검증합니다.

- `listId`는 해당 `unitId`에 속해야 합니다.
- `folderId`는 해당 `unitId`에 속해야 합니다.
- `folderId`와 `listId`는 같은 컨텍스트여야 합니다.
- 조합이 맞지 않으면 `FOLDER_LIST_MISMATCH`로 거부합니다.

Work Graph의 parent 관계는 cycle을 만들 수 없습니다.

- 자기 자신을 parent로 지정할 수 없습니다.
- 자신의 descendant를 parent로 지정할 수 없습니다.
- parent chain 순회는 이미 방문한 노드를 만나면 중단합니다.

### 태스크 CRUD 거버넌스

태스크 CRUD는 사용자 편의보다 데이터 무결성과 추적 가능성을 우선합니다.

Create(생성) 규칙:

- 생성 시 `unitId`, `listId`는 필수이며 `folderId`는 컨텍스트 일치 검증을 통과해야 합니다.
- 생성 직후 최소 추적 단위(`taskId`, 생성자, 생성 시각, 초기 상태)는 타임라인/분석에서 식별 가능해야 합니다.
- FREEFORM 생성은 허용하되, 템플릿 적용 시점에 구조 정합성 검증을 다시 수행합니다.

Read(조회) 규칙:

- 조회 가능성과 편집 가능성은 분리합니다(보이더라도 수정 불가 가능).
- 상세 조회는 parent/children 맥락을 포함하되, 가시 범위 밖 리소스는 누락 또는 차단해야 합니다.
- 관리자 조회는 전체 범위 허용, 일반 사용자 조회는 가시성 규칙(소유/담당/참관/상위 체인)으로 제한합니다.

Update(수정) 규칙:

- 수정은 권한, 무결성(unit/folder/list), 관계(parent cycle), 참조(멘션/노트), 워크플로우 상태를 함께 검증해야 합니다.
- 템플릿 교체 전에는 사용자에게 영향(유지/추가/검토 필요)을 사전 안내하고 확인 후 적용합니다.
- 상태 전이는 `reason`을 필수로 하며 전이 결과는 타임라인과 Inbox에 남겨야 합니다.
- 동시 수정 충돌을 방지하기 위해 낙관적 락(`updatedAt` 기반) 또는 동등한 재시도 정책 도입을 권장합니다.

Delete(삭제) 규칙:

- 삭제는 권한 검증 후 수행하며, 영향 범위(하위 태스크/노트/스레드/타임라인/첨부)를 사전 고지해야 합니다.
- 운영 기본은 soft delete 우선(복구 가능 기간 제공)이며, hard delete는 관리자 승인 경로에서만 허용합니다.
- 삭제 이벤트는 감사 가능하도록 최소 메타(actor, 대상, 시각, 사유)를 기록해야 합니다.

복구/감사 규칙:

- soft delete 대상은 복구 SLA(예: 7~30일) 안에서 원상복구 가능해야 합니다.
- 복구 시 parent chain/권한/템플릿 참조 정합성을 재검증합니다.
- 감사로그는 "누가/무엇을/언제/왜"를 재구성 가능해야 하며 정책 위반 탐지에 활용합니다.

## 4. 뷰와 운영 축

현재 태스크 화면의 주요 뷰는 다음과 같습니다.

- 리스트: 밀도 높은 태스크 조작과 필터링
- 보드: 상태 컬럼 기반 진행 흐름
- 백로그: 우선순위 큐와 계획 전 작업
- 결정 그래프: 계층, 템플릿, 멘션, 결정 참조 관계

## 5. 템플릿과 Form Output

템플릿은 다음을 소유합니다.

- `formDefinition`
- `inspectionCriteria`
- `workflow`
- 선택적 `workflowSchema`

템플릿 적용 시 태스크는 `TEMPLATED`가 되고, `formValues`는 기존 값을 보존한 채 템플릿 필드 키 누락분을 보강합니다. 이후 Form Output UI는 임의 key/value가 아니라 템플릿 필드 정의를 기준으로 렌더링합니다.

추가 운영 규칙:

- 레거시 파일 필드(`__task_files`, `FILE` type)는 저장 시 제거하며 재유입을 허용하지 않습니다.
- `FREEFORM`에서는 블록 단위 Form Output 편집(텍스트/긴텍스트/체크리스트/인용/숫자)을 지원합니다.

### 템플릿 거버넌스 (CRUD / 버전 / 중복)

템플릿은 단순 생성/수정 대상이 아니라 운영 자산으로 관리합니다.

- 템플릿 lifecycle은 `DRAFT`, `ACTIVE`, `DEPRECATED`, `ARCHIVED`를 사용합니다.
- 태스크 상세 템플릿 셀렉터는 `ACTIVE`를 기본 노출 대상으로 하며, `DEPRECATED`/`ARCHIVED`는 기본 숨김합니다.
- 단, 현재 태스크에 연결된 템플릿은 lifecycle과 무관하게 항상 선택지에서 확인 가능해야 합니다.

CRUD 운영 규칙:

- 생성(Create): 기본은 `DRAFT` 또는 `ACTIVE`로 생성 가능하되, 목적(`purposeTag`)과 기대 결과(`successOutcome`)를 함께 관리합니다.
- 수정(Update): 템플릿 본문/워크플로우 변경 시 버전을 증가시키고, 변경 이력을 추적 가능해야 합니다.
- 삭제(Delete): 사용 중 템플릿은 물리 삭제 대신 비활성/수명주기 전환을 우선합니다.
- 조회(Read): 운영 화면(템플릿 센터)은 lifecycle 필터와 검색/정렬을 제공해 과밀 노출을 제어합니다.

버전 관리 원칙:

- 사소한 보정은 기존 템플릿 버전업으로 처리합니다.
- 목적과 구조가 실질적으로 다른 경우에만 새 템플릿을 생성합니다.
- `DEPRECATED`는 신규 선택에서 제외하되, 기존 연결 태스크의 참조 무결성은 보존합니다.

중복 관리 원칙:

- 서버는 템플릿 구조 fingerprint를 계산해 유사 템플릿 후보를 반환할 수 있어야 합니다.
- 유사도 임계치 이상(운영 기본 0.8)의 후보가 있으면 신규 생성보다 버전업/병합을 우선 검토합니다.
- 템플릿 센터는 유사군 신호(예: 유사 N)와 정리 액션(병합/아카이브) 동선을 제공해야 합니다.

## 6. 노트, 댓글, 멘션

댓글은 `mentions` 배열을 가질 수 있습니다.

- `MEMBER`: 멤버 멘션
- `TASK`: 태스크 멘션
- `FORM_FIELD`: 특정 태스크의 템플릿 필드 멘션
- `NOTE`: 노트 참조

작성 UX는 `@`와 `#`를 칩 더미처럼 쌓는 방식이 아니라, 커맨드 검색 메뉴로 후보를 찾고 선택하면 본문 안에 토큰이 삽입되는 방식입니다.
노트는 마크다운 리치 본문과 태그(`tags`)를 함께 관리하며, 노트 링크를 스레드에 붙여넣으면 `#노트` 참조로 자동 변환될 수 있습니다.

검증 규칙은 다음과 같습니다.

- `referencedNoteIds`는 작성자가 볼 수 있는 노트만 허용합니다.
- `TASK`, `FORM_FIELD`, `NOTE` 멘션은 작성자의 가시 범위 안에서만 허용합니다.
- `FORM_FIELD`는 대상 태스크의 템플릿 필드 키가 존재해야 합니다.
- 실패 시 `INVALID_NOTE_REFERENCE` 또는 `INVALID_MENTION`을 반환합니다.

## 7. 상태 전이와 결정

기본 `TaskState`는 다음 네 가지입니다.

- `DRAFT`
- `IN_PROGRESS`
- `DONE`
- `CANCELED`

세부 워크플로우는 `WorkflowStatus`와 `WorkflowStatusCategory`로 표현합니다. `PENDING_APPROVAL` 카테고리는 템플릿 상태 매핑을 위한 분류이고, 실제 승인 대기 런타임은 열린 `ApprovalRequest(status=PENDING)` 존재 여부로 판단합니다.

전이 API는 `reason`을 필수로 받습니다. 승인 없는 일반 상태 전이는 `/api/tasks/:taskId/transitions`, 승인 요청은 `/api/tasks/:taskId/approval-requests`, 승인/반려/보완 판단은 `/api/approval-requests/:approvalRequestId/decisions`로 분리합니다. 결정 타입은 `APPROVE`, `REJECT`, `SUPPLEMENT`, `STATE_ONLY`를 사용하며, 결과는 타임라인과 Inbox에 기록됩니다.

## 8. 승인 정책

승인 정책은 관리자 영역에서 관리합니다.

- 글로벌 정책과 유닛별 정책을 지원합니다.
- 정책은 여러 승인 라인을 가질 수 있습니다.
- 각 라인은 승인자, 조건, 최종 승인자 여부를 가집니다.
- 비활성 정책은 새 결정 흐름에 적용하지 않습니다.

승인 정책은 결정 UX를 구조화하기 위한 운영 규칙이며, 단순 상태 변경과 구분해야 합니다.

승인 요청은 `ApprovalRequest`, 승인/반려/보완 판단은 `ApprovalDecision`으로 기록합니다. 하나의 task에 열린 승인 요청이 있으면 새 승인 요청은 `409 APPROVAL_ALREADY_PENDING`으로 거절합니다.

## 9. Inbox와 알림

서버는 도메인 이벤트를 Inbox 컴포넌트로 분류합니다.

- `DECISION`
- `DISCUSSION`
- `AWARENESS`
- `RESULT`

사용자는 Inbox 항목을 읽음, 확인, 리마인드 처리할 수 있습니다. 알림 설정은 이메일/푸시/데스크톱 채널 및 컴포넌트별 토글을 가집니다.

`read-all`은 관리자여도 현재 사용자 본인의 Inbox만 읽음 처리합니다.

## 10. 분석과 추적

주요 행동은 engagement event로 기록합니다.

- 노드 생성/수정
- 부모 변경
- 템플릿 적용
- Form 저장
- 댓글 작성
- 멘션 생성
- 노트 수정
- 상태/결정 전이

분석 API는 고정값이 아니라 현재 메모리 데이터와 이벤트 로그를 기반으로 지표를 계산합니다.

## 11. 테스트와 변경 관리

아래 규칙은 자동화 테스트로 보호해야 합니다.

- 인증/인가
- IDOR 방지와 가시성
- unit/folder/list 무결성
- 노트 참조와 멘션 검증
- 템플릿 적용과 Form Output
- 상태 전이와 Inbox 라우팅
- 분석 지표 계산

비즈니스 룰을 바꾸면 API 스펙, 시스템 스펙, 테스트, 프론트 안내 문구를 함께 갱신합니다.
