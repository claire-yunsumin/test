# 비즈니스 룰 정리

## 문서 목적

이 문서는 현재 제품에서 반드시 지켜야 하는 핵심 비즈니스 룰을 정리한 기준 문서입니다.  
기획/개발/테스트가 같은 판단 기준을 쓰도록 만드는 것이 목표입니다.

## 1) 사용자/권한 룰

### 역할 계층

- `VIEWER < EDITOR < APPROVER < ADMIN`

### 기본 원칙

- 권한 판단의 최종 책임은 서버에 있습니다.
- 프론트 권한 체크는 UX 안내용이며 보안 경계가 아닙니다.

### 역할별 핵심 정책

- `VIEWER`
  - 가시 범위 내 조회 가능
  - 생성/수정/삭제 제한
- `EDITOR`
  - 태스크/노트/댓글의 일반적인 생성/수정 가능
- `APPROVER`
  - 의사결정 전이(승인/반려/보완) 수행 가능
- `ADMIN`
  - 전체 가시성, 멤버/관리 기능 수행 가능

## 2) 가시성(Visibility) 룰

- 사용자는 "자신이 관련된 태스크" 중심으로 가시 범위를 가집니다.
- `ADMIN`은 전체 태스크를 볼 수 있습니다.
- 일반 사용자는 아래와 연결된 태스크를 볼 수 있습니다.
  - assignee
  - watcher
  - owner
- 태스크 가시성은 상위 계층(parent chain) 맥락까지 포함됩니다.

## 3) 태스크 구조 룰

### 구조 상태

- `FREEFORM`: 자유 형상화 상태
- `TEMPLATED`: 템플릿이 적용된 정형화 상태

### 단위/폴더/리스트 무결성

- 모든 태스크는 `unitId`, `folderId`, `listId` 맥락을 가집니다.
- 생성/수정 시 `folderId`와 `listId` 조합이 일치해야 합니다.
- 불일치 시 요청은 거부됩니다(`FOLDER_LIST_MISMATCH`).

## 4) 협업 룰(노트/댓글/멘션)

### 노트 참조

- `referencedNoteIds`는 "행위자가 볼 수 있는 태스크의 노트"만 허용됩니다.
- 보이지 않는 노트를 참조하면 `INVALID_NOTE_REFERENCE`입니다.

### 멘션

- 멘션 타입: `MEMBER`, `TASK`, `FORM_FIELD`, `NOTE`
- `TASK`/`FORM_FIELD`/`NOTE`는 가시 범위와 정합성 검증을 통과해야 합니다.
- 유효하지 않은 멘션은 `INVALID_MENTION`입니다.

## 5) 상태 전이/결정 룰

- 전이 API에서 `reason`은 필수입니다.
- 결정 타입(`APPROVE`, `REJECT`, `SUPPLEMENT`, `STATE_ONLY`)에 따라 결과 상태가 달라집니다.
- 전이는 타임라인 이벤트로 남아야 하며, 관련 이해관계자에게 Inbox 알림이 라우팅됩니다.

## 6) Inbox 라우팅 룰

- 도메인 이벤트는 서버에서 Inbox 컴포넌트로 분류됩니다.
- 컴포넌트 타입:
  - `DECISION`
  - `DISCUSSION`
  - `AWARENESS`
  - `RESULT`
- 프론트 탭은 서버가 분류한 결과를 표현하는 뷰입니다.

## 7) 감사/추적 룰

- 주요 변경은 `timeline`에 기록되어야 합니다.
- 사용자 참여 이벤트는 `engagement`에 기록되어야 합니다.
- 리텐션/협업 지표는 현재 이벤트/콘텐츠 데이터를 기반으로 계산됩니다.

## 8) 테스트 룰

- 보안/인가/가시성/입력검증 룰은 자동화 테스트로 보호해야 합니다.
- 권한 정책 또는 리소스 가시성 변경 시 테스트를 함께 갱신해야 합니다.

## 변경 관리

- 이 문서의 룰을 변경할 때는 아래를 함께 갱신해야 합니다.
  - API 스펙
  - 보안/인가 테스트
  - 프론트 권한 UX 문구
  - 시스템 스펙(`SYSTEM_SPEC.md`)
