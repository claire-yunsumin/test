# 타임라인 데이터 생애주기 가이드

## 이 문서의 목적

이 문서는 "타임라인 데이터가 어디서 오고, 어떻게 호출되고, 어디로 흘러가서, 화면에 어떻게 보이는지"를 비개발자도 이해할 수 있게 설명합니다.  
코드 공부를 처음 시작할 때 **데이터 흐름 감각**을 잡는 데 초점을 둡니다.

## 한 문장 요약

사용자가 중요한 행동(생성, 수정, 상태 전이 등)을 하면, 서버가 그 행동을 타임라인 이벤트로 기록하고, 프론트가 다시 조회해서 태스크 상세의 타임라인 탭에 보여줍니다.

---

## 1) 데이터는 어디서 시작되나? (발생 지점)

타임라인 데이터는 **사용자 행동**에서 시작됩니다.

대표 행동:
- 태스크 생성
- 태스크 정보 수정(상태/구조/필드)
- 승인/반려/보완 같은 결정 전이
- 노트 생성/수정/삭제

즉, 타임라인은 "자동 생성되는 활동 로그"입니다.

---

## 2) 서버에서 어떻게 만들어지나? (기록 지점)

백엔드에서 각 행동 API 처리 중 `addTimeline(...)`이 호출되며 이벤트가 생성됩니다.

이벤트에 들어가는 핵심 데이터:
- `taskId`: 어떤 태스크의 로그인지
- `type`: 어떤 종류의 이벤트인지 (`TASK_CREATED`, `STATE_TRANSITION` 등)
- `actorId`: 누가 했는지
- `decisionType`: 결정 이벤트인지 여부
- `reason`: 변경/결정 사유
- `referencedNoteIds`: 관련 근거 노트
- `createdAt`: 언제 발생했는지

쉽게 말하면:  
"누가, 언제, 어떤 태스크에서, 무엇을, 어떤 근거로 했는지"를 저장합니다.

---

## 3) 어디에 저장되나? (저장 지점)

현재 프로젝트는 데모 구조라서 DB 대신 인메모리 저장소를 사용합니다.

- 저장 위치: `apps/api/src/domain/store.ts`의 `data.timeline`
- 추가 방식: 최신 이벤트를 앞에 넣는 방식(`unshift`)

나중에 DB로 바뀌어도 개념은 동일합니다.  
저장 매체만 바뀌고, "이벤트를 남긴다"는 규칙은 그대로 유지됩니다.

---

## 4) 프론트는 어떻게 가져오나? (호출 지점)

프론트는 공통 API 함수 `request()`로 서버를 호출합니다.

타임라인을 가져오는 주요 경로:
- 앱 초기 로딩: `GET /api/bootstrap`
- 태스크 상세 진입: `GET /api/tasks/:taskId`

태스크 상세 API 응답 안에 `timeline` 배열이 포함되어 내려오고,  
그 데이터가 상세 화면의 우측 패널(탭)로 전달됩니다.

---

## 5) 화면에서 어떻게 보이나? (표현 지점)

표현 컴포넌트: `TimelinePanel` (`apps/web/src/App.tsx`)

표현 방식:
- 이벤트 타입별 라벨 표시
- 작성자(행위자) 표시
- 결정 타입/사유 표시
- 참조 노트가 있으면 `#노트제목`으로 표시
- 시간 표시(`createdAt`)

추가 UX:
- 필터: 전체 / 시스템 / 변경 / 결정
- 같은 사람의 같은 분(minute) 로그는 묶어서 접기/펼치기
- 펼침 상태는 브라우저 로컬 저장소에 유지

즉, 타임라인은 단순 목록이 아니라  
"빠르게 읽기 좋게 정리된 활동 히스토리 뷰"입니다.

---

## 6) 데이터 흐름 전체 그림

```mermaid
flowchart LR
  A[사용자 행동<br/>생성/수정/전이/노트변경] --> B[API 요청<br/>POST/PATCH/DELETE]
  B --> C[서버 검증<br/>권한/가시성/입력]
  C --> D[addTimeline 호출<br/>이벤트 생성]
  D --> E[data.timeline 저장]
  E --> F[GET /api/tasks/:taskId<br/>또는 /api/bootstrap]
  F --> G[프론트 request() 수신]
  G --> H[TimelinePanel 렌더링<br/>필터/그룹/라벨 표시]
```

---

## 7) 예시로 보는 생애주기

예시: "검토 요청" 버튼을 눌렀을 때

1. 사용자가 전이 요청을 보냄 (`POST /api/tasks/:taskId/transition`)
2. 서버가 권한/입력(`reason`) 검증
3. 서버가 타임라인 이벤트 생성 (`type`, `decisionType`, `reason` 포함)
4. 이벤트가 `timeline`에 저장
5. 프론트가 상세 데이터를 다시 조회
6. 타임라인 탭에 "누가 어떤 사유로 검토 요청했는지" 표시

---

## 8) 왜 이 흐름이 중요한가

- 의사결정 근거가 남아서 나중에 추적 가능
- 팀 협업 시 "무엇이 바뀌었는지"를 빠르게 파악 가능
- 분석/운영 관점에서 행동 데이터를 재활용 가능

한마디로, 타임라인은 단순 로그가 아니라  
팀의 결정 과정을 재구성할 수 있는 핵심 데이터 자산입니다.

---

## 9) 공부할 때 체크포인트

코드 읽기 순서 추천:

1. `apps/api/src/server.ts`에서 `addTimeline(...)` 호출 지점 찾기
2. `apps/api/src/domain/store.ts`에서 `addTimeline` 구현 확인
3. `apps/web/src/lib/api.ts`에서 `request()` 호출 구조 보기
4. `apps/web/src/App.tsx`의 `TimelinePanel`에서 화면 표현 보기

이 순서로 보면 "생성 -> 저장 -> 전달 -> 표시"가 한 번에 연결됩니다.
