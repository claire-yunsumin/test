# Security Operating Checklist

이 문서는 기능별 나열형 체크리스트를 운영 가능한 보안 점검표로 재구성한 버전이다.
핵심은 우선순위, 책임, 주기, 증빙을 명확히 하는 것이다.

## 0) 운영 원칙

1. 보안 점검은 "체크"가 아니라 "증빙"으로 완료한다.
2. 각 항목은 담당자(Owner), 주기(Cadence), 완료 기준(Evidence)을 가진다.
3. 신규 기능 출시 전에는 P0를 모두 통과해야 한다.
4. P1은 스프린트 내 계획, P2는 분기 단위 개선으로 관리한다.

## 1) 우선순위 정의

- P0 (Release Blocker): 미충족 시 출시 금지
- P1 (High): 단기 내 반드시 개선
- P2 (Improve): 운영 성숙도 향상을 위한 개선

## 2) P0 - 출시 차단 항목 (반드시 충족)

### 2.1 인증 / 권한

| 체크 항목 | Owner | Cadence | Evidence |
|---|---|---|---|
| 관리자/내부 라우트(`/admin`, `/dashboard`, `/internal`) 인증 강제 | Backend | 매 릴리스 | 접근 테스트 캡처, 미들웨어 코드 링크 |
| 백엔드 권한 검증(프론트 차단만으로 의존 금지) | Backend | 매 릴리스 | 권한 없는 API 호출 401/403 로그 |
| IDOR 방어(타인 리소스 접근 불가) | Backend | 매 릴리스 | 계정 A/B 교차 테스트 결과 |

## 6) 배포 게이트 (Release Gate)

배포 전 아래를 모두 만족해야 한다.

1. P0 모든 항목 완료
2. 고위험 취약점(High/Critical) 0건 또는 승인된 예외 등록
3. 최근 시크릿 스캔 결과 누출 0건
4. 인증/권한 회귀 테스트 통과
5. 보안 로그/알림 정상 동작 확인

## 7) 빠른 실행 순서 (오늘 바로)

1. 인증/권한 회귀 테스트: 관리자 라우트, IDOR, API 인증 누락
2. 시크릿 누출 점검: 코드, Git 히스토리, 빌드 산출물
3. RLS/데이터 접근 테스트: anon 경로 우회 여부
4. 로그 민감정보 마스킹 점검
5. Rate limit + 비용 알림 + 보안 알림 설정 확인

## 8) 원본 항목 매핑 가이드

기존 77개 체크 항목은 아래로 매핑한다.

- 즉시 사고 가능: P0
- 단기 내 악용 가능: P1
- 성숙도/운영 고도화: P2

필요 시 팀 상황에 맞춰 항목별 Owner와 Cadence를 조정한다.
